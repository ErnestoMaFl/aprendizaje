Lección 04 · Leer CSS

Cascada, especificidad y herencia

Ya lees a qué apunta una regla (Lección 02), qué le cambia (Lección 01) y cuánto espacio ocupa (Lección 03). Falta la pregunta que rompe la cabeza de todo el que abre un CSS ajeno: dos reglas dicen cosas distintas sobre el mismo elemento. ¿Cuál gana? Hoy sales con un método de tres pasos para responderlo siempre.

Antes de leer: arriésgate (fallar aquí es parte del método)

Este <p id="aviso" class="texto"> recibe dos reglas que se contradicen: 

#aviso { color: red; }
.texto { color: blue; }

La regla azul está escrita después. ¿De qué color sale el texto? Apuesta mentalmente — rojo o azul — y sigue leyendo.

El problema: la cascada

CSS significa Cascading Style Sheets: hojas de estilo en cascada. Ese nombre describe justo el mecanismo que vas a aprender hoy. En una web real, decenas de reglas apuntan al mismo elemento. El navegador no se confunde: aplica un orden de desempate, siempre el mismo. Leerlo es lo que te deja decir «este texto sale azul porque…» en vez de encogerte de hombros.

La cadena de decisión, para cada propiedad por separado:

  1. ¿Alguna regla apunta a este elemento? Si ninguna lo hace → puede que herede el valor de su padre (paso 3).
  2. Si varias apuntan → gana la más específica. Aquí es donde importa si usaste #, . o un nombre de elemento. Es el paso decisivo.
  3. ¿Empate exacto en especificidad? → gana la que está escrita más abajo. La última palabra, literalmente.

Léelo en este orden — no al revés

El error típico es mirar primero cuál está escrita al final. El orden solo desempata cuando la especificidad es idéntica. Una regla más específica gana aunque esté escrita 500 líneas antes.

Especificidad: el marcador de tres cifras

Para saber cuál es «más específica», cuenta las piezas del selector y anótalas en un marcador de tres columnas. Se comparan de izquierda a derecha, como un podio: la columna de la izquierda manda sobre todo lo demás.

Con eso ya puedes resolver el duelo del pretest tú mismo. Cuenta los dos marcadores antes de abrir esto:

Ver cómo se resuelve el duelo del principio 
#aviso { color: red; }    /* marcador 1-0-0  ← un id     */
.texto { color: blue; }    /* marcador 0-1-0  ← una clase */
/* Gana 1-0-0: sale ROJO, aunque el azul esté escrito después. */

Un id pesa más que cualquier cantidad de clases

Como las columnas se comparan de izquierda a derecha, #aviso (1-0-0) le gana a .a.b.c.d.e (0-5-0). Una sola cifra a la izquierda decide y ya no se mira el resto: ninguna cantidad de clases «asciende» a la columna de los ids. Por eso # y . no son intercambiables: apuntan distinto y pesan distinto.

Ojo, que esto confunde: dentro de una misma columna las piezas sí se suman — por eso .a.b.c.d.e marca 0-5-0. Lo que no ocurre nunca es que una columna se convierta en otra. Se acumula hacia arriba dentro de cada columna; nunca de una columna a la de al lado.

Herencia: lo que cae del padre solo

Si ninguna regla apunta a un elemento, no siempre se queda «sin estilo»: algunas propiedades las hereda de su elemento padre, automáticamente. Pero solo algunas — y el criterio es más sensato de lo que parece:

Se heredan (cosas de texto)NO se heredan (cosas de caja)
color, font-family, font-size, line-height, text-align margin, padding, border, width, background

La lógica detrás

Piénsalo así: si pones una tipografía en el body, quieres que todo el texto de la página la use — sería absurdo repetirla en cada párrafo. Pero si pones margin: 20px en el body, no quieres que cada párrafo, cada enlace y cada palabra dentro herede 20px de margen: sería un caos. Por eso el texto se hereda; la caja, no. Es justo el modelo de caja de la lección pasada quedándose donde lo pusiste. 

body { color: navy; border: 2px solid red; }

/* Un <p> dentro del body, sin ninguna regla propia:            */
/*   · color  → navy   (heredado: es cosa de texto)             */
/*   · border → nada   (NO heredado: el borde rojo se queda      */
/*                      solo alrededor del body)                 */
La letra pequeña: dónde falla esta heurística (opcional, pero te hará leer mejor)

Dos precisiones que separan «saber la regla» de «entenderla»:

1. Se hereda a los elementos hijos, no «al texto». Una <img> hereda color exactamente igual que un <p> — simplemente no tiene texto que pintar, así que el valor queda ahí, inerte. El texto no hereda nada: no es un elemento, es contenido de un elemento.

2. «El texto se hereda, la caja no» es una correlación, no la ley. Contraejemplos reales, comprobables en MDN:

  • image-rendering — heredable, y solo afecta a imágenes.
  • border-collapse — heredable, y es un borde.
  • cursor, visibility, list-style — heredables, y no son texto.

Lo que de verdad decide si una propiedad se hereda es una pregunta de diseño: ¿tiene sentido escribir esto una vez arriba y que valga para todo lo de abajo? Para font-family, sí. Para image-rendering («escala así todas las imágenes de esta zona»), también. Para padding, sería un caos. Cada propiedad lo lleva declarado en su ficha de MDN como Inherited: yes/no. La heurística te sirve para leer rápido el 80% — ahora sabes por qué funciona y dónde deja de hacerlo.

Periferia: !important y los estilos inline

Dos cosas se saltan el marcador entero. No las verás a menudo en código sano, pero cuando aparezcan tienes que reconocerlas, porque explican resultados que si no parecen imposibles: 

.texto { color: blue !important; }  /* gana aunque el marcador diga que no */

!important pega ese valor al final de la declaración y lo pone por encima de todo lo demás. Y un estilo inline (escrito en el HTML, style="color: green") le gana a cualquier selector de la hoja. Ambos son rompe-reglas: cuando leas un CSS y el marcador no explique lo que ves en pantalla, busca estos dos. (Reconócelos, no los busques como norma — están fuera del 80% del código que leerás.)

Compruébalo — recuerda sin releer

Pregunta 1
Dos reglas chocan: #caja pone color: red, y .caja pone color: blue más abajo. ¿Qué color se ve?

Correcto. #caja marca 1-0-0 y .caja marca 0-1-0. La especificidad decide primero; el orden solo desempata cuando el marcador es idéntico.
El orden de escritura solo importa si hay empate de especificidad. Aquí no lo hay: #caja (1-0-0) le gana a .caja (0-1-0) esté donde esté escrita. Sale rojo.

Pregunta 2
¿Cuál es el marcador de especificidad de .card p.destacado?

Correcto. Dos clases (.card y .destacado) y un elemento (p) → 0-2-1. Ningún id, así que la columna izquierda es 0.
Cuenta pieza por pieza: .card = clase, p = elemento, .destacado = clase. Total: 0 ids, 2 clases, 1 elemento → 0-2-1.

Pregunta 3
En body { font-family: Georgia; padding: 40px; }, ¿qué recibe un <p> de dentro que no tiene reglas propias?

Correcto. font-family es cosa de texto → se hereda. padding es cosa de caja → no se hereda; esos 40px se quedan solo en el body.
Regla de oro: el texto se hereda, la caja no. El p escribe en Georgia (heredado), pero el padding: 40px es del body y de nadie más.

Pregunta 4 · repaso
¿Qué diferencia hay entre #menu y .menu? (de la Lección 02)

Correcto. # = id, único en la página, marcador 1-0-0. . = clase, reutilizable en muchos elementos, marcador 0-1-0. Apuntan distinto y pesan distinto.
Al revés: la almohadilla # señala un id (uno solo en la página); el punto . señala una clase (tantos elementos como quieras). Truco: # es el símbolo de «número», y un número identifica a uno solo.

Pregunta 5
Dos reglas idénticas en especificidad (p y p) piden colores distintos. ¿Quién gana?

Correcto. Con el marcador empatado, manda el orden: la que está más abajo pisa a la de arriba. Es el último paso de la cascada — y solo actúa cuando hay empate.
Con especificidad empatada gana la última escrita — la de más abajo pisa a la de arriba. El valor no influye nunca en quién gana; solo importan especificidad y orden.

Tu misión de lectura (60 segundos)

Aquí tienes tres reglas peleando por el mismo elemento: <a id="inicio" class="boton" href="#">. Calcula los tres marcadores y di de qué color sale — en voz alta, antes de mirar la respuesta: 

a { color: gray; }
#inicio { color: green; }
a.boton { color: orange; }
Pista (si la necesitas)
Cuenta las piezas de cada selector, apunta los tres marcadores y compáralos columna por columna, de izquierda a derecha. No mires el orden en que están escritos todavía.
Ver la respuesta (solo después de escribir la tuya)

a = 0-0-1 · #inicio = 1-0-0 · a.boton = 0-1-1.

Gana #inicio por el 1 de la columna izquierda: sale verde. Fíjate en que está en medio, no al final — el orden aquí da igual, porque no hay empate que desempatar.
