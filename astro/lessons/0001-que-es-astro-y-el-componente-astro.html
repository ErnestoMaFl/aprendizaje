Qué es Astro y cómo se lee un componente .astro

Tu primera victoria: leer un archivo .astro y explicar qué recibe el navegador — y por qué eso hace que el sitio de tu cliente vuele.

🎯 Al terminar podrás: señalar las dos partes de un componente .astro, decir dónde se ejecuta cada una, y explicar en una frase por qué Astro produce sitios más ligeros que un enfoque solo-React.

Primero, un problema (intenta antes de leer)

Un cliente te pide una web corporativa muy rápida. Un compañero te enseña este archivo, que dibuja el titular de la página de inicio:

---
const empresa = 'Panadería Aurora';
const anio = new Date().getFullYear();
---
<h1>{empresa}</h1>
<p>© {anio}</p>

Tu predicción: cuando un visitante abre esta página, ¿cuánto JavaScript descarga y ejecuta su navegador para ver el titular y el año? ¿Se ejecuta ese new Date() en el móvil del visitante o en otro sitio?

Ya lo pensé — muéstrame la respuesta

El navegador descarga cero JavaScript para esto. Astro ejecuta el código de arriba una sola vez, en el servidor o durante el build, calcula empresa y anio, y manda al navegador únicamente el HTML ya resuelto:

<h1>Panadería Aurora</h1>
<p>© 2026</p>

El new Date() corrió en el servidor, no en el móvil del visitante. Si esto fuera un componente típico de React, el navegador descargaría la librería de React más tu componente y volvería a calcularlo en el dispositivo. Esa diferencia es el corazón de la lección.

Equivocarte en la predicción es parte del método: intentar antes de leer hace que la explicación se fije mucho mejor. No te saltes el intento.

Qué es Astro (y para qué NO es)

Astro es un framework web para construir sitios orientados a contenido — blogs, marketing, e-commerce, webs corporativas.1 Justo el tipo de sitio de tu misión. Su apuesta de diseño es «servidor primero»: mueve el trabajo desde el dispositivo del visitante hacia el servidor, y por eso su lema es «menos JavaScript de cliente que ralentice tu sitio».1

La consecuencia práctica se llama cero JS por defecto: tus componentes se renderizan a HTML plano en el servidor, y el navegador no recibe JavaScript salvo que tú lo pidas explícitamente para una parte concreta (eso son las «islas», tema de una lección futura). La documentación reporta cargas «40% más rápidas con 90% menos JavaScript» frente a alternativas cargadas de React.1 Y la velocidad no es estética: más velocidad se traduce en más conversiones para tu cliente.

Contraste con lo que ya conoces. En React, un componente es código que normalmente se envía al navegador y se ejecuta allí. En Astro, un componente .astro es una plantilla que se ejecuta en el servidor y de la que solo sale HTML. Misma palabra —«componente»— idea distinta de dónde corre.

Las dos zonas de un archivo .astro

Todo componente .astro tiene exactamente dos partes.2

---                          // ① SCRIPT: JS que corre en el SERVIDOR
const empresa = 'Panadería Aurora';   //    y se ELIMINA del HTML final
---
<h1>{empresa}</h1>              <!-- ② PLANTILLA: HTML + expresiones {…} -->

① El script del componente va entre las dos líneas de --- (llamadas code fence; a esta zona también se le dice frontmatter). Es JavaScript normal que corre en el build o en el servidor, e importante: «todo será eliminado de la página final enviada al navegador».2 Aquí importas otros componentes, defines variables y obtienes datos.

② La plantilla va debajo del segundo ---. Parece HTML, con extras al estilo JSX: insertas valores con llaves {empresa}, recorres listas con .map() y muestras condicionales con {cond && …}.2

Un componente también puede recibir datos de quien lo usa a través de Astro.props — lo veremos en detalle en la próxima lección, pero conviene que te suene el nombre desde ya.
