Lección 01 · JavaScript del 80%

Guardar un valor: const y let

Es la primera línea de casi todos los archivos .js del mundo y no se puede leer nada sin ella. Vienes de Python, donde guardar un valor es x = 5 y ya está; JavaScript te obliga a elegir una palabra antes del nombre, y esa elección dice algo que Python no puede decir. Al terminar sabrás cuál va, por qué, y —lo importante— qué es exactamente lo que no puede cambiar.

Colores del código en este curso: palabra clave nombre "texto" número función // comentario

Antes de leer: arriésgate (fallar aquí es parte del método)

Dos líneas casi idénticas. Las llaves { } con pares nombre: valor son un objeto: el equivalente al dict de Python. Y const significa «constante». 

const total = 5;
total = 6;                        // (a) ¿funciona o revienta?

const cliente = { nombre: "Ana" };
cliente.nombre = "Luis";       // (b) ¿funciona o revienta?

Apuesta: ¿cuál de las dos da error? ¿Las dos? ¿Ninguna? No lo mires en ningún sitio — arriésgate y escribe por qué. Lo comprobarás tú mismo dentro de un momento.

Primero, cómo ver algo

En Python usas print(). En JavaScript la orden equivalente es console.log(): imprime en la consola —el panel de texto de las herramientas del navegador, o la terminal si usas Node. Lee esta línea como una frase: «del objeto console, usa log con este valor».

const precio = 19.99;
const → cómo se guarda   precio → el nombre   19.99 → el valor

Esa es toda la anatomía de una declaración: palabra clave · nombre · igual · valor. La única pieza que Python no tiene es la primera. El punto y coma final es la costumbre; JavaScript casi siempre lo perdona, pero verás que casi todo el código real lo lleva.

Las dos palabras que se usan

JavaScript moderno tiene exactamente dos formas de declarar, y la diferencia entre ellas es una sola pregunta: ¿este nombre va a apuntar a otro valor más adelante?

PalabraQué prometeCuándo se usa
const núcleo Este nombre apuntará siempre a este mismo valor. Por defecto. La gran mayoría de las declaraciones del código real.
let núcleo Este nombre va a cambiar de valor más adelante. Contadores, acumuladores, algo que se recalcula. Es la excepción.
var solo leer La forma antigua, anterior a 2015. Se comporta de manera más confusa. No la escribas. Recónócela en código viejo y tradúcela mentalmente a let.

La regla práctica que sigue todo el mundo

Empieza siempre con const. Si más adelante necesitas reasignar ese nombre, el propio JavaScript te lo dirá con un error, y entonces —y solo entonces— lo cambias a let. Esto no es manía de estilo: leer const te garantiza de un vistazo que ese nombre no te va a cambiar bajo los pies 40 líneas más abajo. Es información gratis cada vez que lees código ajeno.

Qué es exactamente lo constante

Aquí está el punto que hace tropezar a casi todo el mundo, y la respuesta a tu apuesta. const no congela el valor. Congela el vínculo entre el nombre y el valor.

Piensa en el nombre como una etiqueta pegada a una caja. const significa: «esta etiqueta no se despega de esta caja». No dice nada sobre lo que hay dentro de la caja. Si la caja es un objeto, puedes entrar y cambiar su contenido todo lo que quieras: la etiqueta sigue pegada a la misma caja.

Ver la respuesta del pretest (ábrela solo después de ejecutar)

(b) funciona y (a) revienta con TypeError: Assignment to constant variable.

En (b) no tocas la etiqueta cliente: sigue apuntando al mismo objeto. Solo cambias algo de dentro. En (a) intentas despegar la etiqueta total del 5 y pegarla al 6 — eso es justo lo que const prohíbe.

Y fíjate en el orden de la salida: el console.log(cliente) sí llegó a imprimirse; el error paró el programa después. Los errores de JavaScript detienen la ejecución en el punto exacto donde ocurren.

Cuándo sí necesitas let

Cuando el nombre tiene que apuntar a un valor nuevo. El caso típico es un acumulador o un contador: 

let totalCarrito = 0;
totalCarrito = totalCarrito + 19.99;   // ahora vale 19.99
totalCarrito = totalCarrito + 4.50;    // ahora vale 24.49

Ojo a la segunda línea: no lleva let delante. La palabra clave va solo la primera vez, cuando el nombre nace. Después se reasigna a secas. Repetir let totalCarrito sería declararlo otra vez — y eso también es un error.

solo reconocer  Qué hacer cuando veas var

Muchísimo código publicado antes de 2015 —y aún hoy el 74% de las páginas web, que siguen cargando jQuery— usa var (Web Almanac 2024). Para leer, tradúcelo mentalmente a let y sigue: casi siempre significa lo mismo. Las diferencias finas (dónde vive la variable exactamente) las veremos cuando toque el tema de ámbito. Para escribir: no la uses nunca.

Compruébalo — recuerda sin releer

Pregunta 1
¿Qué ocurre al ejecutar estas dos líneas? 

const colores = { fondo: "azul" };
colores.fondo = "rojo";

Correcto. La etiqueta colores sigue pegada al mismo objeto; solo cambió su contenido. const protege el vínculo, no el interior. Es exactamente el mismo objeto de antes, con otro valor dentro.
Recuerda la caja y la etiqueta: const impide despegar la etiqueta (colores = otraCosa), no impide abrir la caja y cambiar lo de dentro. Como no se reasigna colores, no hay error, y el objeto sigue siendo el mismo — no se crea otro.

Pregunta 2
Necesitas un contador que empieza en 0 y va sumando. ¿Cómo lo declaras?

Correcto. Un contador apunta a un valor nuevo cada vez, así que necesita let. Es justo la excepción que justifica no usar const: la regla es empezar por const y bajar a let solo cuando haga falta reasignar.
Un contador se reasigna (n = n + 1), y eso con const lanza TypeError. Y var es la forma antigua: se reconoce al leer, no se escribe. Queda let, la palabra reservada precisamente para «este nombre va a cambiar».

Pregunta 3 · escribe la respuesta
Vas a declarar el nombre de un cliente, que no cambiará en todo el programa. ¿Con qué palabra clave lo declaras? (una sola palabra)

Correcto: const. Es la opción por defecto y con diferencia la más frecuente en código real. Solo se baja a let cuando el propio JavaScript se queja al reasignar.
La respuesta es const. Si el valor no va a cambiar, declararlo const es gratis y además avisa a quien lea el código —tú, dentro de dos meses— de que ese nombre es estable. let se reserva para lo que sí se reasigna.

Pregunta 4
Lees este fragmento en un proyecto ajeno. ¿Qué te dice de la línea 2? 

let intentos = 0;
intentos = 1;

Correcto. La palabra clave aparece solo cuando el nombre nace. Sin ella, la línea reasigna el nombre que ya existe. Como se declaró con let, reasignar está permitido.
Sin palabra clave delante no hay declaración: es una reasignación del nombre existente. Declarar dos veces sí daría error, pero eso sería let intentos = 1;, con la palabra repetida — no es el caso.

Tu misión de lectura (90 segundos)

Un fragmento como los que te vas a encontrar. Léelo y responde las tres preguntas de debajo sin ejecutarlo: 

const IVA = 0.21;
const pedido = { producto: "Web corporativa", base: 1200 };
let totalFacturado = 0;

pedido.base = 1500;
totalFacturado = pedido.base + pedido.base * IVA;
IVA = 0.10;

a) ¿Cuál de las tres últimas líneas da error, y qué tipo de error?   b) ¿Por qué pedido.base = 1500 está permitido si pedido es const?   c) ¿Cuánto vale totalFacturado justo antes del error?

Pista (si la necesitas)
Para cada línea que empieza sin palabra clave, pregúntate: ¿estoy despegando la etiqueta, o abriendo la caja? Solo lo primero está prohibido. Y para c), recuerda que pedido.base ya cambió una línea antes.
Ver la solución (solo después de escribir la tuya)

a) La última: IVA = 0.10. Es una reasignación de un nombre declarado const → TypeError: Assignment to constant variable.
b) Porque no se reasigna pedido: la etiqueta sigue pegada al mismo objeto. Solo se cambia una propiedad de dentro, y eso const no lo protege.
c) 1500 + 1500 × 0.21 = 1815. La base ya valía 1500 cuando se hizo la cuenta.

Si acertaste las tres, ya lees la parte de un archivo JavaScript que más se repite en todo el código del mundo.
