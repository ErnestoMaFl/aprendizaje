Desenreda la flecha

Diez fragmentos. Ninguno se parece a los ejemplos de la lección: aquí no vale reconocer un patrón, hay que razonar cada uno. Cuatro están rotos, y cada uno lo está de una forma distinta. Predice qué imprime cada console.log antes de tocar la consola del final.

Las dos únicas reglas que necesitas

1. Llaves { } tras la flecha ⇒ bloque ⇒ hace falta return. Sin llaves ⇒ expresión ⇒ se devuelve sola.
2. f es la función; f() es llamarla.

Todo lo demás sale de aplicarlas con cuidado. Si un fragmento te parece imposible, no te falta materia: te falta leerlo más despacio.

Fragmento 1 · calentamiento

const siguiente = x => x + 1;
console.log(siguiente(41));

¿Qué imprime?

Sin llaves ⇒ expresión ⇒ se devuelve sola. 42.
Sin llaves tras la flecha, el cuerpo es una expresión que se devuelve sola: 41 + 1 = 42.

Fragmento 2

const mitad = (x) => { x / 2; };
console.log(mitad(10));

¿Qué imprime?

Correcto: undefined. Hay llaves, así que es un bloque; calcula 5 y lo tira porque falta return. Roto nº 1.
Llaves ⇒ bloque ⇒ hace falta return, y no lo hay. El 5 se calcula y se descarta; la función devuelve undefined sin dar ningún error. Roto nº 1.

Fragmento 3

const suma = (a, b) => a + b;
console.log(suma(7));

Pista de vocabulario: NaN significa «no es un número» — es lo que sale al hacer cuentas con algo que no lo es.

¿Qué imprime?

Correcto: NaN. Falta el segundo argumento, así que b vale undefined, y 7 + undefined no es un número. Ojo: no da error. En Python esto habría reventado con TypeError; JavaScript sigue tan tranquilo. Roto nº 2.
JavaScript no exige que pases todos los argumentos: los que faltan valen undefined. Así que la cuenta es 7 + undefined → NaN («no es un número»), sin ningún error. Ésta es una diferencia grande con Python, que sí habría reventado. Roto nº 2.

Fragmento 4

const etiqueta = texto => { valor: texto };
console.log(etiqueta("oferta"));

¿Qué imprime?

Correcto: undefined. Parece que devuelve un objeto, pero JavaScript lee esas llaves como un bloque, no como un objeto. Y un bloque sin return devuelve undefined. Roto nº 3.
Trampa clásica: las llaves de un objeto y las de un bloque se escriben igual, y JavaScript siempre elige bloque. Así que esto es un bloque sin return → undefined. Roto nº 3.

Fragmento 5 · el mismo, arreglado

const etiqueta = texto => ({ valor: texto });
console.log(etiqueta("oferta"));

¿Qué imprime? (escríbelo tal cual, o «un objeto» si te basta con la idea)

Correcto: el objeto { valor: "oferta" }. Los paréntesis obligan a leer las llaves como un valor, no como un bloque. Un solo carácter de diferencia con el fragmento 4, y resultado completamente distinto.
Aquí sí devuelve el objeto { valor: "oferta" }. Los paréntesis envolviendo las llaves son los que dicen «esto es un valor, no un bloque de instrucciones». Compáralo con el fragmento 4: cambian dos caracteres y cambia todo.

Fragmento 6

const saludar = (saludo = "Hola", nombre) => saludo + ", " + nombre;
console.log(saludar("Ana"));

¿Qué imprime?

Correcto: "Ana, undefined". Los argumentos se reparten por posición, no por sentido: "Ana" cae en el primer hueco, que es saludo. El valor por defecto no se usa (ese hueco sí venía relleno) y nombre se queda vacío. Roto nº 4 — y el más traicionero, porque el error está en el orden de los parámetros, no en la llamada.
Cuidado con la posición: "Ana" es el primer argumento, así que va a parar a saludo, no a nombre. Resultado: "Ana, undefined". Lección práctica: un parámetro con valor por defecto solo es útil si va al final de la lista. Roto nº 4.

Fragmento 7

const doble = n => n * 2;
console.log(doble);

¿Imprime un número o algo distinto? (responde en dos palabras)

Correcto: imprime la función misma, porque faltan los paréntesis que la llaman. En la consola verás algo como ƒ doble(). No está roto del todo — está pidiendo que lo llames.
No hay paréntesis, así que no se llama a nada: se imprime la función como valor (verás algo tipo ƒ doble()). Recuerda la regla 2: doble es la función, doble(4) es el resultado de usarla.

Fragmento 8

const doble = n => n * 2;
const aplicar = (f, v) => f(v);
console.log(aplicar(doble(3), 5));

¿Qué ocurre? (una palabra)

Correcto: TypeError: f is not a function. doble(3) se ejecuta primero y entrega un 6; entonces aplicar intenta hacer 6(5), y un número no se puede llamar. Lo correcto era pasar doble sin paréntesis.
Mira el orden: doble(3) se ejecuta antes de entrar en aplicar, así que lo que se pasa es el número 6. Dentro, f(v) intenta llamar a 6 como si fuera una función → TypeError: f is not a function.

Fragmento 9 · el jefe final

Esta forma no salió en la lección. No hace falta: sale de aplicar las dos reglas dos veces seguidas. Léela muy despacio.

const multiplicador = factor => n => n * factor;

const triple = multiplicador(3);
console.log(triple(4));

¿Qué imprime?

Correcto: 12. multiplicador no tiene llaves, así que devuelve su expresión… y su expresión es otra flecha. Por eso multiplicador(3) no da un número: da una función que recuerda que factor vale 3. Al llamarla con 4 → 4 * 3 = 12. Esto es una fábrica de funciones, y lo verás en código real.
Desenrédala por partes: factor => ( n => n * factor ). La flecha exterior no tiene llaves, así que devuelve lo que hay a su derecha — y ahí hay otra flecha. Entonces multiplicador(3) devuelve una función que multiplica por 3, y triple(4) da 12.

Fragmento 10 · cierre

const resumen = (total) => {
  return { total: total };
};
console.log(resumen(500));

¿Qué imprime? (tal cual, o «un objeto»)

Correcto: { total: 500 }. Aquí las llaves exteriores sí son un bloque, y dentro hay un return explícito con un objeto. Moraleja: el problema del fragmento 4 nunca fue el objeto — era la falta de return.
Funciona: hay llaves (bloque) y hay return, así que devuelve el objeto { total: 500 }. Compara los fragmentos 4, 5 y 10: son las tres maneras de intentar devolver un objeto, y solo la primera falla.
